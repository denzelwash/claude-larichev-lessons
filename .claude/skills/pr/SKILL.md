## Параметры

Скилл принимает необязательные параметры через пробел после `/pr`:

| Параметр | Пример | По умолчанию |
| -------- | ------ | ------------ |
| `title` | `/pr "feat: add auth"` | генерируется автоматически |
| `base-branch` | `/pr "" develop` | `main` |

Пример: `/pr "feat(auth): добавить refresh token" develop`

## Контекст ветки

- текущая ветка: !`git branch --show-current`
- статус: !`git status --short`
- diff stat относительно main: !`git diff main...HEAD --stat`
- коммиты в ветке: !`git log main..HEAD --oneline`

## Правила

PR создаётся через `gh pr create` после пуша ветки в remote.

**Заголовок** — по Conventional Commits: `<type>(<scope>): <описание>` (до 70 символов).

**Описание** обязательно содержит разделы:

```
## Что сделано
- краткие пункты по каждому значимому изменению

## API (если изменялся бэкенд)
- новые или изменённые endpoints с сигнатурой

## Чек-лист для ревью (только для feat/fix/refactor с кодом)
- [ ] типы/DTO в `packages/shared` обновлены
- [ ] фронт и бэк используют типы из `@app/shared`
- [ ] FSD-правило импортов не нарушено
- [ ] новые маршруты добавлены в роутер
```

Для `chore`, `docs`, `ci` и других не-кодовых типов раздел «Чек-лист для ревью» не включать.

Перед созданием PR:

1. Убедиться, что ветка основана на актуальном `main`.
2. Проверить `git diff main...HEAD --stat` — нет ли случайных файлов.
3. Обновить план в `.claude/plans/` (отметить выполненные пункты).

## Алгоритм

1. Изучить блок **Контекст ветки** выше — данные о ветке, diff и коммитах он уже содержит.
2. Определить параметры из аргументов скилла: первый аргумент — `title` (если передан), второй — `base-branch` (если передан, иначе `main`). Для последующих git-команд использовать `base-branch` вместо `main`.
3. Убедиться, что ветка запушена в remote (статус из контекста должен показать `Your branch is up to date with 'origin/...'`). Если нет — `git push -u origin <branch>`.
4. Проверить, что ветка основана на актуальном состоянии целевой ветки: `git fetch`, при отставании — предложить пользователю rebase.
5. Проверить `git diff <base>...HEAD --stat` на случайные файлы (`.env`, билды, дампы) — если есть, остановиться и спросить.
6. Если в репозитории есть план в `.claude/plans/` для этой задачи — открыть, отметить выполненные `- [x]`, закоммитить обновление плана. Если плана нет — пропустить.
7. Определить заголовок PR: если `title` передан как аргумент — использовать его. Иначе сгенерировать по Conventional Commits: `<type>(<scope>): <описание>` (до 70 символов).
8. Собрать тело PR по шаблону: `## Что сделано`, `## API` (если бэкенд менялся), `## Чек-лист для ревью` с обязательными пунктами.
9. Показать пользователю заголовок и тело PR, дождаться подтверждения или правок.
10. Создать PR: `gh pr create --base <base> --title "..." --body "$(cat <<'EOF' ... EOF)"`.
11. Вернуть URL созданного PR из вывода `gh`.
