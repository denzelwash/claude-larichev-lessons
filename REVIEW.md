# Правила ревью

Чек-лист для проверки Pull Request перед мёржем в `main`.

---

## Общее

- [ ] Ветка основана на актуальном `main` (нет расхождения, которое надо решать вручную).
- [ ] В PR нет случайных файлов: `git diff main...HEAD --stat` выглядит разумно.
- [ ] Нет закоммиченных `.env`, секретов, паролей, токенов.
- [ ] Нет `console.log`, `debugger`, временных TODO-комментариев, оставленных как мусор.
- [ ] Заголовок PR следует Conventional Commits: `<type>(<scope>): <описание>` (≤ 70 символов).
- [ ] Описание PR содержит разделы **Что сделано**, **API** (если менялся бэк), **Чек-лист для ревью**.

---

## Shared (`packages/shared`)

- [ ] Новые типы/DTO добавлены в `packages/shared`, а не продублированы в `apps/backend` или `apps/frontend`.
- [ ] Фронт и бэк импортируют общие типы из `@app/shared`, не из локальных файлов.
- [ ] После изменения `packages/shared` оба workspace скомпилированы без ошибок.

---

## Backend (`apps/backend`)

### NestJS

- [ ] Новый функционал оформлен как отдельный модуль (controller / service / module) по аналогии с существующими (`auth`, `categories`, `transactions`).
- [ ] Защищённые маршруты помечены `@UseGuards(JwtAuthGuard)`.
- [ ] Публичные маршруты (если требуются) явно помечены декоратором `@Public()` или аналогом — не снимать guard глобально.
- [ ] DTO использует `class-validator` для валидации входных данных.
- [ ] Контроллер не содержит бизнес-логики — только маршрутизацию и вызов сервиса.

### Prisma

- [ ] Изменения схемы сопровождаются миграцией (`prisma migrate dev`), файл миграции закоммичен.
- [ ] `prisma generate` выполнен после изменения схемы — Prisma Client актуален.
- [ ] Прямых SQL-запросов нет — только Prisma Client API.

### Безопасность

- [ ] Пользователь не может получить или изменить чужие данные (проверка `userId` в запросах).
- [ ] Пароли не возвращаются в ответах API (поле `password` исключено из сериализации).
- [ ] CORS настроен через переменную `CORS_ORIGIN`, захардкоженных origin нет.

---

## Frontend (`apps/frontend`)

### FSD (Feature Slice Design)

- [ ] Слои импортируют только из более низких слоёв: `pages` → `features` → `shared`.
- [ ] Обращение к сегменту (`model/`, `api/`, `ui/`) — только через его публичный `index.ts`, не напрямую.
- [ ] Новая фича добавлена в соответствующий слой, не «свалена» в `shared` или `pages`.

### Vue 3 / TypeScript

- [ ] Компоненты используют `<script setup lang="ts">`.
- [ ] Props и emits типизированы через `defineProps<...>()` / `defineEmits<...>()`.
- [ ] Нет `any` без обоснования; где тип неизвестен — использован `unknown` с явным сужением.
- [ ] Vuetify-компоненты не импортируются вручную (`autoImport: true` покрывает их).
- [ ] Стили: утилитарная разметка (отступы, flex, размеры) — через Tailwind; готовые компоненты — через Vuetify.

### Auth / State

- [ ] Pinia auth store используется для проверки аутентификации, токен не читается напрямую из `localStorage` в компонентах.
- [ ] Защищённые страницы имеют navigation guard в роутере.
- [ ] API-запросы идут через axios-инстанс из `shared/api` (с автоподстановкой токена), а не через `fetch` или новый инстанс.

---

## Производительность и качество

- [ ] Нет избыточных реактивных зависимостей (`ref`/`computed` не оборачивают то, что не нужно реактивности).
- [ ] Нет N+1 запросов к БД — связанные данные загружаются через `include`/`select` в Prisma.
- [ ] Новые маршруты (бэк и фронт) добавлены туда, где они должны быть (router в frontend, controller в backend).
